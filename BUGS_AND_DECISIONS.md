# BUGS & DECISIONS — AI Laboratory / etalhome

> Автоматично зібрано з сесій Claude Code. Мова: українська.
> Останнє оновлення: 2026-05-02
> Джерело: 39 JSONL-файлів сесій, ~750+ діалогових ходів

---

## 🏛 Архітектурні рішення

### 1. Neon PostgreSQL замість in-memory dict і CSV
**Рішення:** Мігрувати всі дані (conversation_histories, stock.csv, price.csv) з ефемерного сховища на Neon PostgreSQL serverless.
**Причина:** Railway при рестарті/деплої втрачає всі дані в пам'яті. Ліди та діалоги зникали після кожного деплою.
**Альтернатива яку відкинули:** SQLite на Railway з Persistent Volume — нестабільно, файл все одно зникає при перебудові контейнера.

### 2. asyncpg + connection pool з max_size=4 для Neon free tier
**Рішення:** `asyncpg.create_pool(min_size=1, max_size=4)` замість psycopg2 або asyncpg без пулу.
**Причина:** Neon free tier обмежує 5 одночасних з'єднань. При `max_size=10` або необмеженому пулі — Connection Exhaustion при паралельних запитах.
**Альтернатива яку відкинули:** psycopg2 (синхронний) — блокує event loop бота при кожному запиті до БД.

### 3. client_id параметризація всіх таблиць БД
**Рішення:** Кожна таблиця містить `client_id TEXT NOT NULL` — universal platform, не etalhome-специфічна.
**Причина:** Платформа мульти-клієнтська. Без `client_id` неможливо ізолювати дані між клієнтами без дублювання таблиць.
**Альтернатива яку відкинули:** Окремі БД на кожного клієнта — занадто дорого і складно управляти.

### 4. Sonnet + tool_use agentic loop замість Haiku + JSON intent enum
**Рішення:** Оркестрант використовує Sonnet з 8 реальними інструментами (`run_audit`, `run_fix`, `run_push`, `run_rollback`, `run_design`, `run_train`, `run_review`, `get_last_url`) і виконує agentic loop до 8 ходів.
**Причина:** Haiku+enum давав "театральні відповіді" — "Агент Website Audit активований. Готовий до SEO-аудиту — надай URL сайту" — без реального виконання. Менеджер думав що агент щось робить, а він просто генерував текст.
**Альтернатива яку відкинули:** Просте розширення enum-переліку — не вирішує проблему fallback, де агент не виконував реальних дій.

### 5. IntentClassifier: перехід від single intent до actions array
**Рішення:** `{"actions": ["audit", "fix", "push"], "url": "...", "confidence": 0.97}` замість одного інтенту.
**Причина:** "Зроби повний SEO для etalhome" — це pipeline із 3 кроків, а не один intent. Підтримка будь-якої комбінації дій без додавання нових прикладів у промпт.
**Альтернатива яку відкинули:** Окремий intent `seo_full` — потрібно вручну додавати кожен новий тип запиту.

### 6. Claude native PDF (document content block) замість pypdf.extract_text()
**Рішення:** Передавати PDF нативно через `document` content block — Claude бачить і текст, і зображення на кожній сторінці.
**Причина:** `pypdf.extract_text()` витягує тільки вбудований текст. Будівельні креслення — скановані зображення всередині PDF, текст порожній → агент відповідав "Не вдалося прочитати PDF".
**Альтернатива яку відкинули:** Google Vision OCR для PDF — вимагає платний billing, складніше в деплої, Claude Vision вже включений в ціну API.

### 7. filters.Document.ALL замість filters.Document.PDF для скріншотів
**Рішення:** Змінити один рядок фільтру: `filters.Document.PDF` → `filters.Document.ALL`.
**Причина:** Скріншот (PNG/JPEG надісланий як файл без стиснення) не матчив `filters.Document.PDF` → бот мовчав. Скріншоти — 90% інформації від Instagram/Telegram клієнтів.
**Альтернатива яку відкинули:** Окремий handler для кожного типу файлу — надмірна складність.

### 8. Збереження відео: кадри через ffmpeg замість нативної підтримки
**Рішення:** Короткострокове — автовідповідь "Надішліть фото/скріншот замість відео". Довгострокове — ffmpeg витягує 3-5 ключових кадрів → передає в Claude Vision.
**Причина:** Claude API не підтримує відео нативно — тільки фото і PDF.
**Альтернатива яку відкинули:** Whisper-транскрипція аудіодоріжки з відео — марно якщо клієнт мовчить у відео.

### 9. Sales Agent = universal framework, специфіка клієнта — в окремому промпті
**Рішення:** Платформа нічого не знає про etalhome. Клієнтська специфіка — в `clients/{client_id}/system_prompt.txt` + `config.yaml`. Placeholder `{{CLIENT_NAME}}`, `{{KB_DATA}}` у шаблоні.
**Причина:** Sales Agent продаватиметься іншим клієнтам, а не тільки etalhome.
**Альтернатива яку відкинули:** Хардкодити etalhome в архітектурі — ускладнює масштабування.

### 10. Google Drive замість S3 для зберігання файлів
**Рішення:** Зберігати оригінали фото/PDF у Google Drive через існуючі service account credentials, URL писати в PostgreSQL.
**Причина:** Service account credentials вже є в проекті (Google Sheets), нуль нових залежностей і платних сервісів.
**Альтернатива яку відкинули:** S3/Cloudflare R2 — нові ключі, нова залежність. Railway Volume — зникає при деплої.
**УВАГА:** Google Drive service account не може писати файли в особисті Drive без Google Workspace (платний). Альтернатива — Telegram Storage (`file_id` в Postgres) + звіт як TEXT у Postgres.

### 11. Fix Agent: Markdown output (Фаза 1) → GitHub PR (Фаза 2)
**Рішення:** Фаза 1 — Fix Agent генерує `.md` з готовими code blocks для ручного вставлення. Фаза 2 — `github_pr.py` читає той самий формат і створює Pull Request.
**Причина:** Блокер — немає GITHUB_TOKEN для репозиторію etalhome.com. Фаза 1 дає цінність зараз без блокера.
**Альтернатива яку відкинули:** Відкласти Fix Agent повністю до Фази 2 — довге очікування.

### 12. FTP Patcher: WordPress mu-plugin замість прямого редагування index.html
**Рішення:** Генерувати mu-plugin (`/wp-content/mu-plugins/`) — WordPress завантажує автоматично без активації.
**Причина:** mu-plugins не потребують активації через WP Admin, автоматично підхоплюються WordPress.
**Альтернатива яку відкинули:** Пряме редагування PHP-файлів теми — ризик зламати активну тему.

### 13. Backup перед кожним /push + /rollback команда
**Рішення:** `ftp_patcher.py` перед `STOR index.html` зберігає `backup_path` в таблицю `fix_history`. Команда `/rollback <url>` відновлює з backup.
**Причина:** Агент патчить index.html напряму через FTP. Якщо щось зламалось — потрібна можливість відкату.
**Альтернатива яку відкинули:** Ручний відкат через FTP клієнт — складно і повільно.

### 14. Gen Trainer: пропозиції → аркуш "Пропозиції", не в FAQ напряму
**Рішення:** trainer.py пише покращення в окремий аркуш "Пропозиції". Роман переглядає вручну → переносить у FAQ.
**Причина:** Якщо trainer перезаписує FAQ напряму — він може внести помилкову інформацію на основі поганих діалогів.
**Альтернатива яку відкинули:** Автозапис в FAQ — ризик деградації Knowledge Base.

### 15. YouTube worker: scan + process в одному скрипті
**Рішення:** Один `youtube_worker.py` = сканування каналів + обробка нових відео. Один Railway cron сервіс.
**Причина:** Два окремих сервіси (Scanner + Processing) вдвічі дорожче. На Railway Trial $3.69 — критично.
**Альтернатива яку відкинули:** Два окремих Railway сервіси — надмірна вартість.

### 16. Haiku для простих повідомлень, Sonnet для складних
**Рішення:** `_is_simple_message()` визначає модель. Haiku для коротких текстових запитів, Sonnet для аналізу зображень і складних питань.
**Причина:** Sonnet в ~10 разів дорожчий за Haiku. Для "Привіт" — Sonnet не потрібен.
**Альтернатива яку відкинули:** Завжди Sonnet — надмірні витрати на прості запити.

### 17. Intent logging в таблицю `intent_logs` (fire-and-forget)
**Рішення:** `log_intent()` у `core/db.py` — asyncio.ensure_future, мовчки ковтає помилки. Не блокує обробку повідомлень.
**Причина:** Потрібна observability — де оркестрант помиляється в класифікації.
**Альтернатива яку відкинули:** Синхронний запис — може заблокувати відповідь клієнту.

### 18. Confirmation gate перед /push у pipeline
**Рішення:** Pipeline `["audit", "fix", "push"]` виконує audit+fix → зупиняється → запитує підтвердження перед деплоєм на продакшн.
**Причина:** Автоматичний деплой на продакшн без підтвердження небезпечний.
**Альтернатива яку відкинули:** Auto-push без підтвердження — ризик поламати живий сайт.

### 19. `is_controversial` flag в dialog_thoughts для ручної перевірки
**Рішення:** `feedback_score` 4-6 → `is_controversial=TRUE`. Дає чергу на ручний перегляд сумнівних відповідей агента.
**Причина:** Потрібна точкова валідація без перегляду всіх діалогів.
**Альтернатива яку відкинули:** Переглядати всі діалоги вручну — нереалістично при масштабі.

### 20. orchestrator.md: заборона фейкових routing-повідомлень
**Рішення:** Явна секція `ЗАБОРОНЕНО` в промпті — заборона "театральних" повідомлень типу "Передаю запит агенту Website Audit".
**Причина:** Оркестрант генерував правдоподібні але порожні відповіді, не виконуючи жодних реальних дій.
**Альтернатива яку відкинули:** Фільтрація на рівні коду — складніше, промпт вирішує проблему в джерелі.

### 21. Sliding window для контексту розмови (8 повідомлень)
**Рішення:** `SLIDING_WINDOW = 8` — агент пам'ятає тільки останні 8 повідомлень активної сесії.
**Причина:** Безмежна історія → переповнення контекстного вікна, зростання вартості API.
**Альтернатива яку відкинули:** Summary + full history — складно, реалізовано як cross-session memory через Haiku summary.

### 23. BUGS_AND_DECISIONS.md як персистентна база знань проекту
**Рішення:** Витягувати знання з JSONL-сесій Claude Code (39 файлів, ~750+ діалогових ходів) у структурований файл `BUGS_AND_DECISIONS.md`. Зберігати в корені `ai-laboratory`. Оновлювати після кожної значущої сесії командою "Додай рішення цієї сесії".
**Причина:** Знання про помилки і рішення губились між сесіями — Claude Code не пам'ятає попередніх розмов. Одні і ті самі помилки повторювались.
**Альтернатива яку відкинули:** Memory-нотатки у `~/.claude/projects/*/memory/` — занадто фрагментовані, не структуровані по темах, важко читати перед початком роботи.

### 24. CLAUDE.md: обов'язкове читання BUGS_AND_DECISIONS.md перед роботою
**Рішення:** Додати в `CLAUDE.md` проекту явну секцію `## ⚠️ Історія рішень і багів — ЧИТАТИ ПЕРЕД РОБОТОЮ` з посиланням на `BUGS_AND_DECISIONS.md`.
**Причина:** Без явної інструкції в CLAUDE.md Claude Code не буде автоматично читати цей файл на початку сесії.
**Альтернатива яку відкинули:** Покластись на memory-систему — memory не містить повного контексту конкретних помилок і рішень.

### 25. ai-laboratory залишається "etalhome Instagram сервером" — не переносити
**Рішення:** Instagram Sales Agent (ai-laboratory) не переносити в ~/etalhome/. Лабораторія є окремим Railway сервісом для Instagram etalhome. ~/etalhome/ = тільки Telegram бот.
**Причина:** Перенесення вимагало б: адаптацію aiohttp → FastAPI, злиття file_handler.py, міграцію env змінних у Railway, ризик downtime при переключенні SendPulse URL. Без переносу — нульовий ризик, нульова робота.
**Альтернатива яку відкинули:** Додати /instagram/webhook endpoint у ~/etalhome/bot/router.py — занадто ризиковано для живого продакшн бота.
**Стратегія масштабування:** Коли з'явиться наступний клієнт — форк ai-laboratory, нові `prompt_template.md` і `DEFAULT_CLIENT_ID`, окремий Railway сервіс.

### 22. Відновлення контексту після рестарту Railway з БД
**Рішення:** При старті діалогу агент завантажує останні 10 повідомлень з `dialogs` таблиці Neon.
**Причина:** Railway перезапускається при кожному деплої — клієнти отримували "перше наше повідомлення" посеред активної розмови.
**Альтернатива яку відкинули:** In-memory dict — зникає при кожному деплої.

## 🐛 Баги та виправлення

### Telegram / Telegram Bot API

| Баг | Причина | Рішення |
|-----|---------|---------|
| Бот мовчав на скріншоти | `filters.Document.PDF` не матчить PNG/JPEG файли | Замінено на `filters.Document.ALL` у `bot/main.py` |
| PDF з кресленнями не читались | `pypdf.extract_text()` повертає порожньо для сканованих PDF | Замінено на нативний `document` content block в Claude API |
| Фото-документи (JPG без стиснення) ігнорувались | Не було роутингу для mime_type image/* у handle_document | Додано перевірку mime_type: зображення → handle_photo_with_addon |
| PTBUserWarning: No JobQueue set up | `python-telegram-bot` встановлений без job-queue екстри | Замінено на `python-telegram-bot[webhooks,job-queue]` у requirements.txt |
| `asyncio.iscoroutinefunction` DeprecationWarning | Python 3.12+ deprecated this function | Замінено на `inspect.iscoroutinefunction()` |
| Telegram HTML-нотифікації з `<` падали | Неекрановані символи `<` у тексті HTML-повідомлення | Додано HTML escaping для символів `<`, `>`, `&` |
| `tool_use_error: Read failed due to the following issue: The parameter 'offset' type is expected as 'number'` | Claude Code надсилав offset як рядок | Системна помилка Claude Code — offset має бути числом |

### Instagram / SendPulse інтеграція

| Баг | Причина | Рішення |

| `extract_image_url()` не знаходила Instagram CDN URL | Regex не покривав `cdninstagram.com`, `fbsbx.com` | Оновлено regex: Instagram/Facebook CDN + URL що закінчується на .jpg/.png/.gif/.webp |
| Деякі повідомлення проходили мимо webhook | "One-time" флоу закривався після першої відповіді | Флоу переконфігурований на "постійний" режим |

| Трапецієвидна перегородка (30/33) → невірна ціна | Правило ×1.4 для трапецієвидних не було задокументовано | Додано у prompt: трапецієвидна перегородка → ×1.4 до площі |
| Розміри в мм → агент не конвертував у метри | Явного кроку конвертації не було у промпті | Додано: "Спочатку конвертуй мм → м (1200 → 1.2м) до будь-яких обчислень" |
| Перегородка з дверима → окремий тариф на двері | Агент рахував перегородку + двері окремо | Правило: "Перегородка з дверима = 1 виріб, рахуємо за тарифом перегородки по загальній площі" |

### Оркестрант

## ⚠️ Уроки — що НЕ робити

- **Не оновлювати webhook URL в SendPulse до деплою нового endpoint** — якщо поміняти URL раніше ніж задеплоїти новий сервіс, повідомлення клієнтів летять на неіснуючий endpoint → downtime. Порядок: деплой → перевірити що endpoint відповідає → тільки потім змінювати SendPulse URL.

- **Не рахувати підтвердження деплою як явну команду** — "зроби" або "так" без "пушимо"/"деплой" = не деплоїти. Завжди чекати явну команду.

- **Не читати файли проєкту для планування** — працювати з контекстом розмови і memory. Читання файлів для аналізу без необхідності спалює токени і час.

- **Не починати повний рефакторинг core-файлів без явної команди** — "фейкові відповіді" ≠ "вся архітектура неправильна". Мінімальний точковий фікс першим, повний рерайт тільки після прямого "перепиши все".

- **Не давати Роману завдання які Claude може виконати сам** — міграції, seed-скрипти, команди з .env — виконувати самостійно.

- **Не передавати history з Neon напряму в Anthropic API** — Neon зберігає `ts` і `meta`, Anthropic приймає тільки `role` і `content`. Завжди стрипати перед API-викликом.

- **Не ігнорувати Railway Trial баланс** — кожен новий cron-сервіс коштує гроші навіть в idle. Перед деплоєм нових сервісів — перевірити баланс і апгрейднути план.

- **Не зберігати файли локально на Railway** — будь-який файл (SQLite, CSV, logs) зникне при наступному деплої. Тільки зовнішня БД (Neon) або сховище (Google Drive).

- **Не використовувати `pypdf.extract_text()` для сканованих PDF** — для будівельних креслень потрібен Claude Vision через нативний `document` content block.

- **Не хардкодити etalhome в архітектурі агентів** — Sales Agent = universal framework. Специфіка клієнта в `clients/{client_id}/` папці, не в коді.

- **Не запускати trainer.py напряму в FAQ** — тільки через аркуш "Пропозиції" з ручним переглядом. Автозапис може деградувати Knowledge Base.

- **Не залишати `confidence: 0.9` захардкоджений** — Instagram webhook мав `"confidence": 0.9` для всіх повідомлень, тренер не міг знайти слабкі місця.

- **Не будувати новий функціонал поки підключення до живого бота не перевірено** — блоки 1-4 архітектури були побудовані але не підключені до живого бота. Треба спочатку підключати, потім розширювати.

- **Не ставити `filters.Document.PDF` якщо обробляти будь-які файли** — скріншоти PNG/JPEG надіслані "як файл" (без стиснення) не матчать PDF фільтр.

- **Не ігнорувати mode оператора в SendPulse** — відкритий чат оператора = бот не відповідає. Завжди закривати чат після ручної відповіді.

- **Не змішувати etalhome і ai-laboratory репозиторії** — це два окремих проекти. Перевіряти правильний repo до першого читання файлів.

- **Не шукати ціни в коді проєкту** — прайс etalhome живе ТІЛЬКИ в Google Sheets KB (`fileId: 1EELiSfK3tA_k3KxEjEte-VHQFPUDfMuTGEfzOBMsyC4`, аркуш `Prices`). Перевіряти через MCP `mcp__claude_ai_Google_Drive__read_file_content`. Локально через gspread — помилка 403 (service account немає прав). В коді, БД (`knowledge_base`) і `.env` цін немає.

- **Не рахувати площу з фото якщо є текстові розміри** — агент розрахував повну площу П-подібної душової кабіни (~4.2 м²) з креслення замість конкретних розмірів дверей (700×2000=1.4 м²), що спричинило завищення ціни на 5 400 грн. Текстові розміри клієнта завжди мають пріоритет над фото/кресленням.

- **Не вважати RAL фарбування рами нестандартним** — порошкове фарбування профілю по RAL (будь-який колір, включно RAL 9010) — стандартна послуга etalhome. Агент помилково ескалював до менеджера. Нестандартне скло (флутс, матове, тоноване) — ескалювати, рама — ні.

- **Не приймати "Litestream" при наявності PostgreSQL** — якщо PostgreSQL вже підключено, Litestream — дублювання рішення без цінності.

- **Не будувати YouTube → Sales без перевірки чи є канал взагалі** — `yt_summaries` порожній якщо немає відео на каналі. Перевіряти наявність контенту до реалізації інтеграції.

- **Не деплоїти `asyncpg` без перевірки requirements.txt** — при імпорті `from core import db` у новий файл Railway перебудовує контейнер і падає якщо asyncpg відсутній.
