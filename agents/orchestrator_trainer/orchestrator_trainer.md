# Orchestrator Trainer

**agent_id:** *(не є агентом з agent_id — це scheduled analyzer)*  
**Модель:** `claude-sonnet-4-6`  
**Запуск:** щоденно (cron/Railway scheduler)

## Призначення

Аналізує діалоги менеджера з оркестрантом за останні N годин, знаходить помилки в роботі оркестранта і пропонує покращений system prompt. Зміни не застосовуються автоматично — чекають на підтвердження менеджера в Telegram.

## Пайплайн

1. `db.load_history(client_id, MANAGER_TELEGRAM_ID, "telegram", limit=100)` → завантаження діалогів
2. Фільтрація за останні N годин (за `ts` полем)
3. `db.get_agent_prompt(client_id, "orchestrator")` → поточний промпт
4. Claude аналізує: поточний промпт + діалоги → JSON з issues + new_prompt
5. `db.save_pending_review()` → зберігає як `pending` (ще не активний)
6. Менеджер отримує Telegram-сповіщення з issues та change_log
7. Після `/approve` → новий промпт стає активним

## Вихід аналізу

```json
{
  "issues": ["опис проблеми 1", "..."],
  "change_log": ["Додано правило X", "..."],
  "new_prompt": "повний текст нового промпту"
}
```

## Золоті правила (НЕЗМІННІ — analyzer їх не змінює)

1. Мова відповіді: українська
2. Правила підтвердження деструктивних дій (push/rollback)
3. Список доступних інструментів та їх назви
4. Правила ескалації до менеджера
5. Заборона виконувати дії без явного підтвердження

## Skip умови

| Умова | Дія |
|-------|-----|
| `MANAGER_TELEGRAM_ID` не задано | `skip` |
| Немає повідомлень за N годин | `skip` |
| Немає промпту в БД | `skip` (auto-seed при наступному рестарті) |

## Env vars

| Змінна | Опис |
|--------|------|
| `MANAGER_TELEGRAM_ID` | Telegram ID менеджера |
